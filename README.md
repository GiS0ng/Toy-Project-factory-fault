 스마트팩토리 설비 진동 모니터링 및 이원화 저장 시스템
1. 시스템 개요 (System Overview)
    본 시스템은 Edge단(공장 현장 컴퓨터)에서 1호기 설비의 센서 데이터를 실시간으로 수집하고, 다중 조건문 알고리즘을 통해 설비의 이상 징후를 판정하는 C++ 기반의 임베디드 모니터링 시스템입니다.

    단순 감시에 그치지 않고 데이터 유실 방지(Data Integrity)를 위한 임시 버퍼링(Buffering) 메커니즘과 오작동 필터링(Fault Tolerance) 누적 카운터를 탑재하여 시스템의 견고함을 극대화했습니다.

2. 데이터 흐름 및 이원화 저장 아키텍처
데이터의 성격과 활용 목적에 따라 파일 저장 시스템을 완전히 분리하는 이원화 구조를 채택했습니다.

① 정기 가동 일지 (Periodic Telemetry Log)
대상 데이터: NORMAL(정상) 및 WARNING(경고) 데이터

저장 메커니즘: 매 순간 파일에 쓰지 않고, 메모리 바구니(vector<VibrationLog> periodic_buffer)에 데이터를 임시로 모아둡니다.

주기: 30분 단위 (시뮬레이션상 30초)로 묶어서 일괄 파일로 출력(Bulk Write)합니다.

파일명 포맷: YYYYMMDD_HHMMSS_1호기_INTERVAL_30M.csv

목적: 전체 가동 트렌드 분석 및 향후 파이썬/DB 연동을 통한 예지보전 AI 학습용 데이터 확보.

② 비상 블랙박스 로그 (Event-driven Alarm Log)
대상 데이터: 설비 셧다운을 유발한 치명적 데이터

저장 메커니즘: 위험 상황 누적 한계 도달 시 주기를 기다리지 않고 즉시 파일로 덤프(Dump)합니다.

생성 조건: 16회 이상 비상 진동이 총 4회 누적되었을 때 발동.

파일명 포맷: YYYYMMDD_HHMMSS_1호기_CRITICAL_누적4회오류.csv

목적: 설비 정지 원인 추적 및 즉각적인 사후 대처를 위한 하드웨어 블랙박스 역할.

3. 핵심 알고리즘 및 예외 처리 로직
① 4회 누적 셧다운 (Fault Tolerance Counter)
순간적인 센서 노이즈나 일시적 외부 충격으로 장비가 꺼지는 오작동을 막기 위해 누적 Threshold 알고리즘을 적용했습니다.

진동수 16회 이상 발생 시 즉시 종료하지 않고 내부 카운터(critical_counter)를 1씩 증가시킵니다.

카운터가 마침내 4에 도달했을 때만 치명적 고장으로 확정하고 셧다운 프로세스를 밟습니다.

② 크래시 직전 데이터 대피 (Pre-Crash Data Save)
30분이 채 되기 전에 4회 누적 에러로 시스템이 강제 종료(exit(0))될 경우, 메모리 바구니에 담겨있던 "종료 직전까지의 정상 데이터"가 증발하는 문제를 해결했습니다.

강제 종료 직전, 버퍼에 데이터가 있다면 YYYYMMDD_HHMMSS_1호기_INTERVAL_30M_이전데이터.csv 파일로 긴급 대피 저장을 수행한 후 시스템을 셧다운합니다.

4. 데이터 포맷 정의 (CSV)
파이썬이 데이터를 쪼개어 읽기(Parsing) 가장 좋고, 향후 SQLite DB에 적재하기 가장 용이한 쉼표(,) 구분자 포맷을 사용합니다.

포맷 형식: [시간], [호기번호], [총 진동수], [에러코드]

데이터 예시:
`2026-06-19 09:45:02, 1, 3, 0      -> 정상 운전 데이터
2026-06-19 09:45:05, 1, 12, 101   -> 육안 점검 필요 경고
2026-06-19 09:45:11, 1, 26, 102   -> 4회 누적을 유발한 치명적 비상`

5. 향후 확장 스펙 (파이썬 & DB 연동 계획)
이 C++ 프로그램이 완성됨에 따라 차후 단계에서 구현할 파이썬(Python) 파이프라인은 다음과 같습니다.

    1. 파일 와쳐 (File Watcher): 파이썬 프로그램이 프로젝트 폴더를 실시간 감시합니다.

    2.분류기 (Parser): 파일명 끝자리가 INTERVAL_30M이면 통계 테이블로, CRITICAL이면 비상 알림 테이블로 분기 처리합니다.

    3.내장형 DB 적재 (SQLite): sqlite3 라이브러리를 활용하여 가볍고 빠르게 데이터를 테이블에 적재하고, 최종적으로 원격 대형 DB(PostgreSQL 등)로의 마이그레이션 구조를 설계합니다.
